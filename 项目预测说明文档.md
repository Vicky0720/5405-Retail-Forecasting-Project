## 2. 任务定义

根据课程项目要求，本次任务定义如下：

- 预测目标：`option sales quantity`
- 预测粒度：`option_id × day`
- 预测 horizon：未来 28 天
- 当前首周预测的 feature date：`2022-07-14`
- 预测区间：`2022-07-15` 至 `2022-08-11`
- 提交格式：每个 `option_id` 输出长度为 28 的浮点数数组

因此，这是一个典型的多时间序列销量预测问题，同时具有以下特点：

- 存在层级关系：`option -> item -> cate`
- 明显的周内季节性
- 销量分布不均，头部长尾差异显著
- 一部分 option 较为稀疏，常出现零销量
- 价格、折扣、活动、上下线状态都可能影响销量

## 5. 数据处理方式

### 5.1 历史销量标准化

原始销量表进入模型前，先做以下标准化处理：

1. 将 `date` 转为日期类型
2. 自动识别销量字段
3. 按 `option_id + date` 聚合销量
4. 构建完整的 `option_id × date` 索引
5. 缺失日期补零

这样得到统一的历史面板：

- `option_id`
- `date`
- `target`

补零的目的不是“假设没卖就是缺数据”，而是把建模粒度固定成完整日频面板，方便后续 lag 和 rolling 特征计算。

### 5.2 原始业务面板保留

除了标准化历史面板外，脚本还额外保留了一个 `raw_daily_panel`，用于保存每天的原始业务信息：

- `sales_amount`
- `label_amount`
- `is_online_begin`
- `is_online_end`
- `player_group_id`

这样做的原因是：

- 预测目标只需要 `target`
- 但价格、折扣、上下线状态不能在标准化时丢掉

### 5.3 维表与事件拼接

在进入机器学习特征工程之前，会把历史面板与以下信息拼接：

- `options`
- `items`
- `events`
- `raw_daily_panel`

从而得到包含层级、价格、折扣、事件、上下线状态的统一训练面板。

## 6. 特征工程

当前版本的特征工程可分为 8 类。

### 6.1 日期特征

用于刻画基本日历季节性：

- `dow`
- `dom`
- `month_num`
- `weekofyear`
- `is_weekend`
- `trend_idx`

### 6.2 option 自身时序特征

用于刻画 option 自身销量惯性：

- `lag_1`
- `lag_7`
- `lag_14`
- `lag_21`
- `lag_28`
- `lag_35`
- `lag_42`
- `lag_56`

### 6.3 滚动统计特征

用于刻画近期水平与稀疏度：

- `roll_mean_7`
- `roll_mean_14`
- `roll_mean_28`
- `roll_mean_56`
- `roll_zero_7`
- `roll_zero_14`
- `roll_zero_28`
- `roll_zero_56`

其中：

- `roll_mean_*` 表示最近窗口平均销量
- `roll_zero_*` 表示最近窗口中零销量比例

### 6.4 item 层级特征

先按 `item × date` 聚合得到 item 总销量，再构造：

- `item_target`
- `item_lag_1`
- `item_lag_7`
- `item_lag_14`
- `item_lag_28`
- `item_roll_mean_7`
- `item_roll_mean_28`
- `option_share_in_item`

这些特征用于捕捉：

- item 整体热度变化
- option 在 item 内部的份额关系

### 6.5 价格与折扣特征

当前版本已经显式引入价格与折扣特征，包括：

- `unit_sales_price`
- `unit_label_price`
- `discount_ratio`
- `discount_depth`
- `price_gap_to_label`
- `relative_price_in_item`
- `price_rank_in_item`

解释如下：

- `unit_sales_price`：实际成交单价
- `unit_label_price`：标价单价
- `discount_ratio`：成交价 / 标价
- `discount_depth`：折扣深度
- `price_gap_to_label`：成交价与标价差异
- `relative_price_in_item`：相对 item 内均价的位置
- `price_rank_in_item`：在同 item 选项中的价格排序

这些特征的目标是帮助模型学习价格弹性和同 item 内的相对替代效应。

### 6.6 上下线状态特征

当前版本也引入了 option 生命周期相关特征：

- `online_state`
- `online_state_change`
- `days_since_online_begin`

解释如下：

- `online_state`：当前是否处于在线阶段
- `online_state_change`：最近是否发生上下线状态变化
- `days_since_online_begin`：距离最近上线起点已过去多少天

这些特征用于帮助模型识别：

- 新上线初期的爬坡
- 下线前后的异常销售模式

### 6.7 活动强度与事件特征

当前引入：

- `event_name`
- `event_type`
- `item_promo_intensity`

其中 `item_promo_intensity` 用于近似表示某个 item 当前所处的促销活跃程度。

### 6.8 分群特征

为了区分高销量 option 与稀疏 option，当前脚本构造：

- `vol56`
- `nz56`
- `segment`

其中：

- `vol56`：最近 56 天总销量
- `nz56`：最近 56 天非零率
- `segment`：按照当日横截面分组得到的类别

分群规则为：

- `high_volume`
  - `vol56 >= 当日中位数`
  - 且 `nz56 >= 0.45`
- `sparse`
  - `vol56 <= 当日 25% 分位数`
  - 或 `nz56 <= 0.20`
- 其他为 `mid`

这个分群结果直接用于树模型的分群训练。

## 7. 使用的预测方法

当前系统不是单一模型，而是 4 路候选模型加上两层集成逻辑。

### 7.1 方法总览

候选模型包括：

1. `wbaseline`
2. `etr`
3. `lgbm`
4. `item_topdown`

最终有两种集成输出：

- `ensemble_selected`
  - 使用全局统一权重
- `ensemble_bucket_selected`
  - 按 horizon bucket 使用不同权重

当前正式预测实际采用的是：

- `ensemble_bucket_selected`

因为它允许不同预测阶段使用不同模型组合。

## 8. 各算法如何进行预测

### 8.1 `wbaseline` 统计基线

这是一个按 `option_id` 单独建模的统计预测器。

其预测逻辑为：

1. 用最近 `7 / 28 / 56` 天销量均值做加权平均，得到基础水平
2. 用星期几均值构造 weekday seasonality
3. 比较最近 14 天与前 14 天，估计短期趋势
4. 对趋势进行截断，避免外推过猛
5. 对稀疏 option 做收缩，减少零销量长尾的过预测

可以写成近似形式：

`forecast = level × seasonal_ratio × damped_trend × sparse_shrinkage`

优点：

- 稳定
- 可解释
- 对样本很少的序列较安全

缺点：

- 难以学习复杂非线性关系
- 不能充分利用价格、折扣和层级信息

### 8.2 `etr`：ExtraTrees 全局 direct multi-horizon 模型

`ExtraTreesRegressor` 属于随机化树集成模型。

当前用法不是“一个模型递归往后滚”，而是：

- 针对每个 horizon `h = 1...28`
- 各训练一个 direct model

也就是说，模型会直接学习：

- “今天往后第 1 天怎么预测”
- “今天往后第 7 天怎么预测”
- “今天往后第 21 天怎么预测”

这比递归预测更适合长 horizon，因为不会把前一天预测误差层层传递下去。

`etr` 的训练数据来自所有 option 拼接后的全局面板，因此它能同时学习：

- 个体滞后关系
- item 层级变化
- 价格与折扣效应
- 节日效应
- 稀疏度差异

同时，当前版本不是统一训练，而是按 `segment` 分群训练：

- `high_volume`
- `mid`
- `sparse`

若某一群当前样本量不足，则回退到全量样本训练，避免分群导致训练不稳。

当前 `etr` 参数经过进一步调优，主要方向是：

- 增加树数量
- 控制叶子最小样本数
- 限制 `max_features`

这使得模型在稳定性与表达能力之间取得更好的平衡。

### 8.3 `lgbm`：LightGBM 全局 direct multi-horizon 模型

`LightGBM` 也是树模型，但属于梯度提升树，与 `ExtraTrees` 的学习方式不同。

它的核心思想是：

- 每一轮新树都去拟合上一轮残差
- 逐步逼近真实目标

当前同样采用 direct multi-horizon：

- 每个 horizon 单独训练一个模型

并使用与 `etr` 基本一致的输入特征。

当前 `lgbm` 的参数设计偏保守和正则化，核心设置包括：

- `objective = poisson`
- 较小 `num_leaves`
- 较高 `min_child_samples`
- `max_depth = 6`
- `reg_lambda`
- `reg_alpha`
- `subsample`
- `colsample_bytree`
- `bagging_fraction`

使用 `poisson` 的原因是：

- 销量本质上是非负计数
- 对偏右分布、长尾销量通常更自然

`lgbm` 的优势是：

- 更擅长学习复杂非线性和交互项
- 对价格、活动和层级特征通常更敏感

不足是：

- 若调得过强，容易对最近一段历史过拟合
- 当前样本量下，整体上还没有稳定超过 `etr`

### 8.4 `item_topdown`：先 item 后 option 的层级预测

这个方法不直接在 option 层预测，而是分两步：

1. 先在 `item` 层做 28 天预测
2. 再把 item 预测值按照近期 option share 分配回各个 option

item 层预测又是一个组合：

- `baseline`
- `ETS`

其中 `ETS` 来自 `ExponentialSmoothing`，核心思想是：

- 用指数平滑方式分解 level、trend、seasonality
- 最近信息权重大，较旧信息权重递减

这个方案特别适合：

- item 总体比较稳定
- option 内部份额变化不剧烈

优点：

- 对头部 item 较稳
- 能利用层级结构减少纯 option 级噪声

缺点：

- 若 item 内 share 波动很大，拆分误差会被放大

### 8.5 整体权重集成

脚本先在 `selection` 窗口上搜索 4 路候选模型的最优全局权重，得到：

- `selected_weights`

当前结果为：

- `wbaseline = 0.0`
- `etr = 1.0`
- `lgbm = 0.0`
- `item_topdown = 0.0`

说明在整体平均意义下，当前最稳定的全局最优模型就是 `etr`。

### 8.6 按 horizon bucket 集成

考虑到不同预测阶段可能适合不同模型，脚本进一步把 28 天拆成 4 个 bucket：

- `1-7`
- `8-14`
- `15-21`
- `22-28`

然后在 `selection` 上分别搜索每个 bucket 的最优权重。

当前得到的 bucket 权重为：

- `1-7`
  - `etr = 0.7`
  - `item_topdown = 0.3`
- `8-14`
  - `etr = 1.0`
- `15-21`
  - `etr = 0.9`
  - `lgbm = 0.1`
- `22-28`
  - `etr = 1.0`

这说明：

- 短期 `1-7` 天，item 层级信息还有边际价值
- 中后段 `15-21` 天，LightGBM 能带来少量补充
- 其余阶段 ExtraTrees 最稳

最终 week 1 正式预测采用的正是这套 bucket 权重。

## 9. 回测设计

### 9.1 为什么不能只看单次回测

如果只用一个 feature date 做回测，结论很容易偶然化。

所以当前版本采用多轮 rolling origin backtest。

### 9.2 当前回测方案

当前默认配置为：

- 回测 origin 总数：`10`
- holdout origin 数：`2`

具体 origin 为：

- `2022-04-14`
- `2022-04-21`
- `2022-04-28`
- `2022-05-05`
- `2022-05-12`
- `2022-05-19`
- `2022-05-26`
- `2022-06-02`
- `2022-06-09`
- `2022-06-16`

其中：

- 前 8 个 origin 作为 `selection`
- 后 2 个 origin 作为 `holdout`

### 9.3 三段式验证思想

当前流程的核心目的是避免“在同一批回测窗口上既调权重又宣称泛化效果”。

因此流程是：

1. 用历史数据训练候选模型
2. 在 `selection` origins 上比较模型并搜索最优权重
3. 在完全未参与调权重的 `holdout` origins 上评估泛化能力

这比“单层回测直接选最优”更健康，也更接近真实未来表现评估。

### 9.4 回测输出

当前会输出三类评估文件：

1. `backtest_scores`
   - 整体分数
2. `backtest_metrics_by_horizon`
   - 每个 horizon 的分数
3. `backtest_metrics_by_bucket`
   - 每个 horizon bucket 的分数

并且三者都同时保留：

- 逐 origin 明细
- `ALL_SELECTION`
- `ALL_HOLDOUT`

这样既能看局部波动，也能看聚合结论。

## 10. 结果评估方法

### 10.1 使用指标

当前使用的核心指标是：

- 按真实销量加权的 `1-MAPE`

即：

对于每个 `option-day` 观测，先计算：

`1 - |y - y_hat| / y`

再按真实销量 `y` 做加权平均。

### 10.2 为什么使用这个指标

该指标的含义是：

- 预测越接近真实销量，分数越高
- 高销量样本权重更高

这与零售场景更一致，因为：

- 主力 option 的预测误差通常比长尾零销量误差更重要

### 10.3 当前指标解读

在当前脚本输出中：

- 分数越高越好
- `0` 附近表示很差
- 越接近 `1` 越好

因此，`0.55` 明显优于 `0.45`。

### 11.6 按 horizon 的 holdout 观察

从单日 horizon 的最优模型分布来看：

- 部分 horizon 上 `etr` 最优
- 部分 horizon 上 `lgbm` 最优
- 部分 horizon 上 `item_topdown` 最优
- 少数 horizon 上 `wbaseline` 仍能胜出

这正说明：

- 不同 horizon 的最佳模型并不相同
- 按 bucket 分配权重是合理的

## 12. 当前结果有效性

### 12.1 为什么说当前结果是“有效的”

当前结果有效，主要基于以下几点。

#### 1. 没有未来信息泄漏

每个 backtest origin 都只使用该日期之前的数据训练，再预测未来 28 天。

#### 2. 采用了独立 holdout

当前最重要的一点是：

- 权重只在 `selection` 窗口上搜索
- `holdout` 窗口不参与调权重

因此 holdout 结果更接近“外推能力”。

#### 3. 引入了更合理的业务特征

当前模型已经利用了：

- 价格
- 折扣
- 上下线状态
- item 内份额
- 活动信息
- 稀疏度信息

相比只用简单滞后项，更有现实业务意义。

#### 4. 对头部与长尾做了区别处理

通过 `segment` 分群，模型不再默认高销量和稀疏 option 用同一种训练方式。

### 12.2 为什么当前结果仍然不能视为“最终最优”

虽然当前方案已经比早期版本更稳，但仍存在以下限制：

#### 1. holdout 轮数仍然不算很多

当前只有 2 个 holdout origins，虽然比单次验证好，但仍不足以证明绝对稳健。

#### 2. bucket 仍是离散分段

现在是按 `1-7 / 8-14 / 15-21 / 22-28` 分 bucket。

更细的做法可能是：

- 按单 horizon 自动选模型
- 或按 horizon 动态平滑权重

#### 3. LightGBM 还没有做更系统参数搜索

当前参数已经优于默认配置，但仍属于经验调优，而非完整搜索。

#### 4. 稀疏 option 仍有改进空间

虽然当前已分群，但对极度稀疏 option，规则型约束可能还需要更强。

## 13. 当前方案的结论

综合当前代码、回测与 holdout 结果，可以给出如下结论：

1. 当前最稳定的整体单模型是 `ExtraTrees`
2. 价格、折扣、上下线状态、item 层级与事件特征确实值得保留
3. 高销量与稀疏 option 采用分群建模是合理方向
4. bucket 粒度集成比单一统一权重略优，说明不同 horizon 的最佳模型并不一致
5. 当前结果具有现实参考价值，适合作为第一周正式提交方案
6. 但若要继续冲分，仍应进一步强化 horizon-level gating、参数搜索和更长窗口验证

## 14. 后续可继续改进的方向

1. 把 bucket 权重进一步升级为 horizon 级别自动 gating
2. 对 `etr` 与 `lgbm` 做更系统的参数搜索
3. 增强价格弹性和促销记忆特征
4. 对极度稀疏 option 增加更强的零销量保护
5. 引入更长时间范围的 holdout 检验
6. 如果环境允许，可继续尝试更强的全局 boosting 方案或更细的层级 reconciliation
